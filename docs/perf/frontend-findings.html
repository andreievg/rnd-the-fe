Frontend exploration — findings & recommendations

Building the Stocktake page with SolidJS: data layer, tables, and virtualization — what we tried, what it cost, and what to pick.

📦 Repo: github.com/andreievg/rnd-the-fe  ·  all docs  ·  raw metrics

I spent this exploration building one real screen — the Stocktake table — several different ways, measuring each under identical conditions, and comparing bundle size and load performance. This document summarises the branches, the numbers, and the reasoning behind the recommendations.

1.What was done

Starting from a SolidJS + webpack base with vanilla-extract CSS, I layered on one concern at a time: a hand-rolled GraphQL layer (with a codegen plugin, test-covered), then a real table — first via TanStack Table, then hand-rolled, and finally a heavier data stack using TanStack Query + a GraphQL client with a codegen-generated SDK.

Each stage is its own branch on the fork (andreievg/rnd-the-fe):

  • solid SolidJS + webpack starter (baseline shell, no data).
  • base-graphql Table + vanilla-extract + hand-rolled GraphQL layer (graphqlFetch), renders all rows (no virtualization).
  • tan-stack-table TanStack Table (v8) + solid-virtual — virtualized.
  • own-backed-table Hand-rolled virtualized table (solid-js only, no table library).
  • tan-stack-query-and-graphql-client Heavier stack: TanStack Query + GraphQL client + codegen-generated SDK.
  • tests-for-plugin The codegen plugin, validated with tests.
  • own-table-no-virtualisation-editable Editable list + a zero-reactivity static-HTML baseline (static-baseline/index.html) — the "do we even need virtualization?" test.

Full per-run metrics (5 runs each, warm-up dropped, median + spread) live in docs/perf/frontend-runs.html on main.

Reading the table: the GraphQL branches all fetch the same 1506-row payload in one request, so TTNetworkQuiet is ~500 ms everywhere — that floor is one request's latency, unbeatable by a table/client choice. TTDataRendered is where the approaches diverge. "Rows" is mounted DOM rows: 1506 for the non-virtualized branch, ~20–26 for the virtualized ones (dataset is 1506 for all). † load and LCP are recorded in the full doc but are not ranked on — for an SPA they fire long before the data the user came for is on screen.

Headline result. Virtualization is the single biggest lever: rendering all 1506 rows costs 1400 ms and 27k DOM nodes; virtualizing collapses render out of the critical path. Among the virtualized options the hand-rolled table is fastest and lightest (655 ms, 17 KB, 3.4 MB) — but see §5 for why TanStack Table is still the recommendation.

3.On SolidJS

I like Solid over React. State is easier to reason about and it's noticeably more performant, while keeping a familiar React-like look and feel. We don't need zustand + selectors for state — Solid's fine-grained reactivity handles it — and it ships primitives (like createResource) that make hand-rolling a GraphQL layer straightforward.

The trade-off is learning: the team would need to get comfortable with Solid's reactivity model. Claude can help with that, and there's a primer here → andreievg.github.io/learn-solidjs.

4.The GraphQL layer — hand-roll or adopt?

The first "build vs buy" decision is the data layer. The surprising finding: the heavier off-the-shelf stack (TanStack Query + graphql-request + generated SDK) shows both a big bundle increase (43 KB vs 17 KB over-wire) and a performance hit — TTDataRendered 1030 ms vs 655 ms for the same table and same single request. The extra ~375 ms is the client's parse/execute/wrapper overhead on the critical path under throttle.

Why the hit, when caching should help? This is a cold first-load of a read-once page — it never exercises Query's caching, dedup, or background-refetch benefits, so here they're pure cost. On a screen that refetches, reuses cache across navigations, or runs many queries, that calculus changes. We're measuring the worst case for a caching layer.

We can hand-roll the layer instead. A codegen plugin generates typed operations, and we can validate it with tests — see the tests-for-plugin branch. Calling the generated client (no cache) looks like this: 

const data = await graphqlFetch(StocktakeLines, {
  stocktakeId: STOCKTAKE_ID,
  storeId: STORE_ID,
  page: { offset: state.rows.length, first: CHUNK },
  sort: [{ key: 'itemName', desc: false }],
});

It's easy to wrap this with createResource into a global resource that exposes a refetch method plus a condition to only render on the first refetch (called in onMount). That gives us the ergonomics we actually need without the bundle and startup cost of the full client.

Recommendation. Hand-roll the GraphQL layer with a tested codegen plugin + a createResource wrapper. Reach for TanStack Query only on screens that genuinely need cross-view caching / refetch orchestration.

5.Tables & virtualization

Hand-rolling a table is easy for the base case — and, as the numbers show, it's the fastest and lightest. But complexity grows fast as features are added, and the biggest one is virtualization, which we absolutely need.

I tried several ways around building it: growing the list with a "loading as you go" approach, and building virtualization from scratch. In practice it's a genuinely complex task — think about prepending rows when scrolling up, having to measure them, then adjusting scroll position to compensate, and so on.

"Maybe we don't need virtualization"?

I tested that assumption directly. With an editable list, rendering all rows is slow — and crucially, it's not a Solid or reactivity problem: I stripped it down to a completely static HTML page with zero reactivity and it was still slow. Open the static baseline → (source lives on the own-table-no-virtualisation-editable branch). The conclusion: the cost is the sheer DOM size (27k nodes for 1506 rows), so virtualization is non-negotiable for large tables regardless of framework.

So: adopt TanStack Table

Given we need virtualization and a pile of other features (sorting, filtering, column sizing, selection…) that a hand-rolled table would have to grow into, TanStack Table is worth its bundle size. And the v9 Solid version is catered more to Solid with advanced tree-shaking — the spike already showed ~15% smaller bundle (29.8 vs 34.7 KB gzip) at speed parity, and it should improve further as it stabilises. v9 is beta

Recommendation. Use TanStack Table (target v9 for Solid once stable) for virtualization + a mature feature set. The hand-rolled table wins on raw numbers but loses on maintenance the moment we need real table features.

6.Bonus takeaway — chunked loading for big tables

For large tables, query for just ~100 rows first, then immediately fetch the rest and append to the list. This shows the table to the user quickly (first paint on a small slice) while the full dataset streams in behind it — a much better perceived-performance story than waiting on the whole 1506-row payload before rendering anything.
